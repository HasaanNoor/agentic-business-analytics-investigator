# Project Overview

## Goal

Create a local Python application that monitors a simulated ecommerce business called Northstar Commerce.

The application generates realistic synthetic business data, detects unusual KPI changes, groups related anomalies into business incidents, predicts future business performance, explains why those predictions were made, retrieves similar historical incidents, generates actionable reports, and exposes the generated outputs through a local FastAPI API.

## Inputs

- No real customer data.
- Synthetic data generated by the project.
- Local CSV, JSON, Markdown, model, figure, and pickle files created by earlier pipeline steps.

## Outputs

- Synthetic datasets in `data/synthetic/`.
- Report files in `outputs/reports/`.
- Figure files in `outputs/figures/`.
- Forecast models in `outputs/models/`.
- Local RAG knowledge base in `outputs/rag/`.
- FastAPI endpoints that read existing output files.

## Current Project Scale

The current implementation includes:

- 3 years of deterministic synthetic ecommerce data (2024–2026)
- Approximately 210 enriched business incidents generated from a deterministic incident catalog rather than a fixed interval.
- 210 multi-agent investigation reports
- 420 searchable RAG knowledge records
- More than 50 automated tests

## Main files to create

- `README.md`
- `requirements.txt`
- `src/` package folders for each major component of the application.
- `tests/` files for the pipeline and API.

## Required behavior

- The project must run locally.
- The pipeline must be deterministic enough for repeatable tests.
- Each phase must be runnable as its own script.
- Each phase must read files created by previous phases and write its own outputs.
- The project should use simple, descriptive names.
- The project should not require a paid service for normal local use.
- Optional narrative writing may use an API key if one is available, but a deterministic fallback report must work without one.

## Acceptance criteria

- A new developer can understand the project from the README.
The pipeline order is clear:

1. Generate synthetic data.
2. Validate data.
3. Build daily KPI summary.
4. Detect anomalies.
5. Group anomalies into incidents.
6. Train forecasting models.
7. Generate forecasts.
8. Explain forecasts.
9. Write actionable reports.
10. Run the multi-agent investigation workflow.
11. Build a searchable knowledge base from historical incidents.
12. Retrieve similar historical incidents to improve recommendations.
13. Serve generated outputs through a read-only FastAPI API.
- Tests cover each major phase.
